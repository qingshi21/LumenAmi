# Live2D 自动化生成与情感驱动 — 可行性分析与落地计划

## 一、架构总览

```
用户入口层 → 模型生成层 → 参数识别与动作生成层 → 实时驱动层 → 用户最终体验
```

目标闭环：用户提供角色描述/图片 → 自动生成 Live2D 模型 → AI 识别参数并生成动作 → 对话中根据情感实时驱动表情动作变化。

---

## 二、逐层可行性分析

### 2.1 用户入口层

| 输入方式 | 可行性 | 说明 |
|---------|--------|------|
| 文本描述 | ✅ 完全可行 | 纯前端表单，无技术障碍 |
| 图片上传 | ✅ 完全可行 | 前端上传 + 后端存储，成熟方案 |

**结论：无风险，直接实现。**

---

### 2.2 模型生成层

| 工具 | 可行性 | 风险 | 说明 |
|------|--------|------|------|
| Textoon（文本→Live2D） | ⚠️ 可行但质量不稳定 | 中 | 生成的模型参数完整度参差不齐，物理模拟和参数定义质量需人工校验 |
| live2d-automation（图片→Live2D） | ⚠️ 可行但精细度不足 | 中 | 自动拆图层和参数绑定能做到"能跑"，但头发飘动、衣服摆动等细节物理往往不够 |

**输出产物：**
- `.moc3` 模型数据
- `.model3.json` 配置
- 贴图 `.png`
- 参数定义 `.cdi3.json`
- 物理模拟 `.physics3.json`

**关键风险点：**
1. 生成的 `.moc3` 版本必须匹配 Cubism 4 SDK（当前项目使用 `pixi-live2d-display/cubism4`）
2. 贴图路径和格式需符合规范，否则模型加载透明（已有踩坑经验）
3. 参数 ID 命名是否标准化，直接影响后续动作生成的参数映射

**结论：技术上可行，但 AI 生成模型的质量波动较大，短期内建议手动选模型兜底，自动化生成作为远期目标。**

---

### 2.3 参数识别与动作生成层

| 环节 | 可行性 | 风险 | 说明 |
|------|--------|------|------|
| analyze_model.py 读取参数列表 | ✅ 完全可行 | 低 | `.cdi3.json` 是标准 JSON，解析简单 |
| AI 代理编写 `.motion3.json` | ⚠️ 理论可行，实际不稳定 | **高** | `.motion3.json` 是时间序列关键帧格式，LLM 对数值序列生成能力有限 |
| validate_motions.py 校验 | ✅ 完全可行 | 低 | 对照参数范围做规则校验，纯工程问题 |

**核心瓶颈：LLM 生成高质量 `.motion3.json` 的可靠性**

`.motion3.json` 的结构要求每个参数在不同时间点给出精确的数值关键帧，LLM 容易出现：
- 参数值超出合法范围（`.cdi3.json` 定义的 min/max）
- 时间轴关键帧不连贯，动作抖动
- 多参数之间不协调（如开心时嘴角上扬但眉毛下垂）

**推荐策略：模板库 + AI 选择，而非 AI 从零生成**
- 预置 10-15 个标准情绪动作模板（开心、悲伤、惊讶、生气、害羞等）
- AI 根据对话情感选择最匹配的模板触发
- 后续可探索 AI 对模板参数做微调，但需严格校验

**结论：纯 AI 生成动作文件的路线风险较高，建议走"预置模板 + AI 选择"的混合路线。**

### 2.4 实时驱动层

#### 已有能力（基于当前代码）

| 能力 | 状态 | 实现位置 |
|------|------|----------|
| 模型加载 | ✅ 已实现 | `renderer.js` — `Live2DModel.from()` |
| 动作播放 + 优先级 | ✅ 已实现 | `playMotion()` — FORCE/IDLE/NORMAL 三级优先级 |
| 口型同步 | ✅ 已实现 | IPC + 正弦波驱动 `ParamMouthOpenY` |
| 鼠标跟随 | ✅ 已实现 | `focus()` + 平滑插值 + 全局追踪 |
| 待机循环 | ✅ 已实现 | `scheduleIdleMotion()` — 随机间隔播放 Idle 组 |
| 滚轮缩放 | ✅ 已实现 | scale 控制 + 窗口尺寸同步 + 防抖 |

#### 需要新增的能力

| 能力 | 难度 | 说明 |
|------|------|------|
| 情感标签解析 | 低 | 在 system prompt 中要求 AI 输出情感标签，后端正则提取 |
| 情感→动作/表情映射 | 中 | 建立映射表，将情感标签转为 motion 调用或参数设置 |
| 表情与口型叠加 | 中 | 说话时口型动画和表情动画需同时生效，需管理参数优先级 |

**情感标签方案：**
```
AI 回复格式示例：[emotion:happy] 哈哈好的，那我们去吧！
后端解析：正则匹配 [emotion:xxx]，提取情感标签
IPC 传递：emotion-start { emotion: "happy", duration: 3000 }
前端映射：happy → playMotion("Happy", 0, "NORMAL") 或设置表情参数
```

**参数叠加问题：**
当前口型动画直接调用 `setParameterValueById('ParamMouthOpenY', value)`，如果表情也走参数直设，需要确保：
- 口型优先级 > 表情（说话时嘴部以口型为准）
- 表情结束时有淡出过渡，不突兀归零
- 眼部/眉毛参数不与口型参数冲突

**结论：基础设施完善，情感驱动是增量工作，技术风险低。**

## 三、整体可行性评估

| 层 | 可行性 | 风险等级 | 备注 |
|----|--------|----------|------|
| 用户入口 | 100% | 无 | 纯 UI 功能 |
| 模型生成 | 80% | 中 | AI 生成模型质量不稳定，需人工兜底 |
| 参数识别与动作生成 | 60% | 高 | AI 生成 `.motion3.json` 可靠性是最大瓶颈 |
| 实时驱动 | 85% | 低 | 基础设施已有，情感映射是增量开发 |
| 端到端闭环 | 70% | 中 | 跑通没问题，体验调优需要持续投入 |

**总体判断：架构技术上全部可行，没有不可逾越的障碍。最大不确定性在"AI 自动生成动作文件"环节。**

## 四、落地计划

### Task 1：情感标签驱动（优先级最高，投入产出比最大）

**目标：** 对话中 AI 回复带情感标签，宠物实时做出对应表情/动作

**实现步骤：**
1. 修改后端 system prompt，要求 AI 在回复开头输出 `[emotion:xxx]` 标签
2. 后端 `ChatService` 增加正则解析逻辑，提取情感标签
3. 通过 WebSocket/IPC 将情感标签传递给前端悬浮窗
4. 前端建立情感→动作映射表（初期映射到已有 motion group）
5. 处理表情与口型动画的参数叠加和优先级

**预计工作量：** 2-3 天

---

### Task 2：情绪动作模板库建设

**目标：** 预置一套标准情绪动作，供情感驱动调用

**实现步骤：**
1. 梳理需要支持的情绪列表（开心、悲伤、惊讶、生气、害羞、困惑、思考等）
2. 为每种情绪制作/收集 `.motion3.json` 动作文件
3. 定义情绪动作的存放规范（目录结构、命名规则）
4. 前端增加按情绪名称加载和播放动作的能力
5. 可选：为每种情绪定义 `.exp3.json` 表情文件（眼部/眉毛等参数组合）

**预计工作量：** 3-5 天（取决于情绪种类数量和动作精细度）

---

### Task 3：模型参数自动识别工具

**目标：** 自动读取任意 Live2D 模型的参数列表，为后续动作生成做准备

**实现步骤：**
1. 编写 Python 脚本解析 `.cdi3.json`，提取所有参数 ID、名称、取值范围
2. 输出标准化的参数清单（JSON 格式）
3. 可选：提供 Web UI 展示参数列表，方便调试

**预计工作量：** 1 天

---

### Task 4：AI 辅助动作生成（探索性）

**目标：** 探索让 AI 根据参数清单生成简单动作文件的可行性

**实现步骤：**
1. 设计 prompt，将参数清单 + 目标情绪描述喂给 AI
2. 让 AI 输出 `.motion3.json` 格式的动作数据
3. 编写校验脚本，检查参数值是否合法、时间轴是否合理
4. 人工审核 AI 生成的动作，评估质量
5. 根据评估结果决定是直接使用还是仅作为辅助参考

**注意：** 此任务为探索性质，效果可能不稳定，建议在前三个 Task 完成后再推进

**预计工作量：** 2-3 天（探索 + 评估）

---

### Task 5：文本/图片 → Live2D 模型自动生成（远期）

**目标：** 用户输入描述或上传图片，自动生成可用的 Live2D 模型

**实现步骤：**
1. 调研并集成 Textoon / live2d-automation 等工具
2. 搭建模型生成 pipeline（输入 → 生成 → 校验 → 输出）
3. 增加模型质量自动校验（检查 `.moc3` 版本、贴图完整性、参数合法性）
4. 提供人工审核/替换环节（生成后让用户预览确认）
5. 对接到现有的模型加载流程

**前置条件：** Task 1-3 已完成，基础体验已跑通

**预计工作量：** 5-7 天（含调试和兜底逻辑）

---

## 五、推荐落地顺序

```
Task 1（情感标签驱动）  ← 最快见效，体验提升最大
   ↓
Task 2（情绪动作模板库）← 为 Task 1 提供更丰富的表现力
   ↓
Task 3（参数自动识别）  ← 为后续自动化打基础
   ↓
Task 4（AI 辅助动作生成）← 探索性，视效果决定投入深度
   ↓
Task 5（模型自动生成）  ← 远期目标，依赖工具链成熟度
```

**核心原则：先用手动模型 + 预置模板跑通全链路体验，再逐步用 AI 自动化替换各环节。**
