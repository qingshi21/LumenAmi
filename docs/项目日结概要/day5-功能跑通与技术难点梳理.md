## 遭遇的核心困难与思考

### 1. UI 视觉不够商业化

当前玻璃拟态风格虽然统一了暗色基调，但整体质感仍停留在"个人项目"水平，缺乏商业产品的视觉吸引力。主要问题：

- 卡片层次不够丰富，缺少精致的阴影和光效细节
- 动效体系不完整（hover/active 有，但缺少入场动画、过渡转场）
- 宠物形象区域太空洞，没有实际的视觉内容（只有 emoji）
- 缺少品牌感的视觉元素（图标体系、插画、微动效）

**方向**：需要引入更精细的设计系统，参考成熟产品（如 Character.AI、Replika）的视觉语言。


### 2. AI 调用的长期记忆问题

当前每次调用 AI 的流程是：

```
最近10条聊天记录 + 全部 memory → 拼接成 prompt → 发给 AI
```

这存在几个严重隐患：

- **memory 只增不减**：随着对话积累，memory 会越来越多，prompt 越来越大，AI 响应越来越慢
- **没有记忆衰减机制**：真实的人类记忆会淡化，但系统里所有 memory 权重相同
- **每次都是"新的 AI"**：AI 本身没有持久记忆，完全依赖每次拼接的 prompt，本质上是"伪装成长期记忆的短期上下文"
- **prompt 膨胀是时间问题**：总有一天 prompt 会大到超出 token 限制或导致响应异常缓慢

**需要的方案**：

- memory 需要有"重要度衰减"或"定期归档"机制，自动清理低价值记忆
- 探索向量数据库（如 ChromaDB）做记忆检索，只取相关记忆而非全量
- 或者引入"记忆摘要"层：把旧记忆压缩成摘要，只保留关键信息
- 参考分层上下文管理架构，用固定层，检索层，上下文层优化AI回复速度
- 理想状态：AI 有真正的"长期记忆"能力，而不是每次都靠 prompt 伪装


### 3. 宠物只有聊天，缺乏视觉表现

目前宠物在前端只是一个 emoji + 聊天气泡，本质上是一个"带 AI 的角色扮演聊天工具"，缺乏桌宠产品最核心的"陪伴感"视觉呈现。

**核心问题**：

- 宠物数据目前只存储了 `name`、`role`（角色设定）、`petId`，没有外观/动作相关的数据结构
- 前端悬浮窗需要宠物"会动"，而且要根据对话内容产生反应（如赞同时点头、开心时摇摆）
- 即使不手动设计每个动作，能否让 AI 自己输出动作指令？比如 AI 回复时附带一个 `action` 字段

**技术路径思考**：

- 方案 A：使用 Lottie/Spine 动画，预设一套动作库，AI 回复时指定动作 ID
- 方案 B：使用 Live2D 模型，更生动但开发成本高
- 方案 C：AI 输出时附带表情/动作标签，前端根据标签播放对应动画
- 需要先确定宠物形象的存储格式（图片序列？骨骼动画？CSS 动画？）


### 4. AI Prompt 与角色扮演的平衡

当前 System Prompt 包含：用户信息 + 记忆 + 角色设定 + 对话历史。问题在于：

- **表演过度**：AI 可能过度"扮演"角色，说一些不自然的话
- **缺乏人物特色**：有时 AI 回复完全没有角色应有的性格
- **陪伴感与角色设定的矛盾**：比如达米安·德斯蒙斯这种傲娇角色，需要有反骨的性格但同时给用户陪伴感，这个平衡很难通过 prompt 精确控制

**举例**：创建"艾莎"角色，用户期望的是从她这里获取陪伴感，但 AI 可能输出不知所云的内容。成熟角色还好，性格复杂的角色（傲娇、反骨）更难把握。

**方向**：

- 优化 System Prompt 的结构，把"陪伴感"作为底层要求，角色性格作为表层特征
- 可能需要 few-shot examples 来引导 AI 理解角色的说话方式
- 考虑在 prompt 中明确"底线规则"：无论什么性格，核心是让用户感到被关心


### 5. 用户安全认证

当前登录状态只存在 `localStorage` 的 `userId` 和 `token` 中，没有做真正的安全认证。

**计划方向**：

- JWT（JSON Web Token）做无状态认证，token 过期自动刷新
- bcrypt 做密码加密存储（当前后端可能还是明文存储）
- 需要考虑 token 刷新机制和登出处理


### 6. 屏幕截取能力

豆包等产品已经有屏幕感知相关的研究，LumenAmi 也希望具备这个能力。

**涉及问题**：

- Electron 的 `desktopCapturer` API 可以实现屏幕截取
- 需要用户授权和权限管理（隐私合规）
- 截取后的图像如何传给 AI 分析？（多模态模型 or OCR）
- 使用场景：宠物"看到"用户在做什么，主动发起话题


### 7. 社区功能（远期）

未来可能开放社区市集，用户可以分享和下载现成的 AI 桌宠预设（角色设定 + 外观 + 声音）。

**涉及问题**：

- 需要一个内容审核机制
- 角色预设的数据格式标准化
- 可能需要后端支持文件上传和社区 API


## 补充思考

### 性能与成本

- 每次 WebSocket 消息都走后端转发，高并发时需要考虑连接管理
- Qwen API 调用有成本，memory 全量注入会导致 token 消耗持续增长
- 需要监控 API 调用量和 token 消耗，设置合理的上限

### 数据持久化

- 聊天记录目前存在数据库，但历史消息的清理策略还没有
- 需要考虑用户数据备份和迁移

### 多端一致性

- 目前只有 Windows 桌面端，未来是否考虑 Mac / 移动端
- Electron 本身支持跨平台，但 UI 适配需要额外工作

## 总结

Day 5 的核心收获是：**基础功能跑通只是起点，真正的挑战在于"质量"和"体验"**。

技术上最大的瓶颈是 AI 记忆机制 — 当前方案是"用 prompt 伪装长期记忆"，这条路有明确的上限，需要尽早引入真正的记忆管理方案。

产品上最大的缺口是"宠物不会动" — 桌宠的核心价值是"陪伴感"，而陪伴感很大程度上依赖视觉表现。一个只会聊天的宠物和一个会点头、会摇摆、会根据对话情绪反应的宠物，体验差距是巨大的。

接下来需要在"AI 记忆优化"和"宠物视觉表现"两个方向上重点突破。
